# Пользователь приложения

## Назначение

Пользователь приложения является общей учетной записью Pocket. После регистрации аккаунт может использовать гостевой функционал, создать рабочее пространство владельца или получить приглашение сотрудника.

## Доступный функционал

| Область | Возможности |
| --- | --- |
| Регистрация | E-mail, пароль, имя, согласие с правилами и подтверждение адреса |
| Вход | E-mail и пароль, восстановление доступа и управление сессией |
| Роли | Переключение между гостем, владельцем и сотрудником при наличии соответствующих прав |
| Аккаунт | Личные данные, телефон, пароль, язык, уведомления и выход |
| Каталог | Просмотр всех опубликованных заведений независимо от рабочей роли |

## Жизненный цикл роли

1. Новый аккаунт получает роль гостя по умолчанию.
2. После создания первого заведения backend добавляет роль владельца рабочего пространства.
3. После принятия приглашения пользователь получает роль сотрудника конкретного заведения.
4. Переключение роли меняет интерфейс, но не создает отдельную учетную запись.
5. Удаление доступа сотрудника отзывает только venue membership и не удаляет личный аккаунт.

## Авторизация

- Access token используется для API-запросов и имеет короткий срок действия.
- Refresh token хранится в защищенной HttpOnly cookie и ротируется.
- Backend возвращает список memberships: роль, `workspaceId`, `venueId` и permissions.
- Frontend восстанавливает последнюю допустимую роль, но всегда обрабатывает отзыв прав со стороны backend.
- Критичные изменения аккаунта требуют повторной аутентификации.

## Интеграция

- `POST /auth/register`, `POST /auth/login`, `POST /auth/refresh` и `POST /auth/logout`.
- `GET /me` возвращает профиль и доступные роли.
- `PATCH /me` обновляет личные данные.
- Подтверждение e-mail и восстановление пароля выполняются одноразовыми токенами.
- OAuth можно добавить после MVP без изменения модели ролей и memberships.
